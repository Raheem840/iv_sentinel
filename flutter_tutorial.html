What is IV Sentinel?

IV Sentinel is a Flutter mobile app that polls ThingSpeak IoT channels — one channel per hospital bed — and shows nurses a live dashboard of fluid levels. When any bed crosses into CRITICAL the phone vibrates and fires a local push notification.

📦 State
Riverpod 2 — reactive global state management. Replaces setState + InheritedWidget.
🌐 Data
ThingSpeak REST API, polled every 15s. Pure HTTP — no Firebase, no WebSockets.

Chapter 1

Widgets — Everything Is a Widget

In Flutter, literally everything on screen is a widget: a button, a text label, the padding around a button, even the app itself. Widgets compose — you nest them inside each other exactly like HTML elements.

Key Concept: Widget

A Widget is a description of part of a UI. It's immutable — once built, it doesn't change. When data changes, Flutter builds a new widget and compares it to the old one to figure out what to repaint (this is called the "diff").

StatelessWidget vs StatefulWidget

There are two kinds of widgets. The rule is simple:

Widget TypeUse when…IV Sentinel example
StatelessWidgetThe widget only depends on its inputs (props). Same inputs → same output, always.StatusBadge, SparklineChart
StatefulWidgetThe widget has internal state that changes over time (e.g. an animation, a timer).BedCard, CriticalBanner

A Real StatelessWidget — StatusBadge

StatusBadge shows a colored pill: "NORMAL", "LOW", or "CRITICAL". It takes a statusCode integer and a small flag. Same inputs → same look, always. No internal state needed.

lib/features/home/widgets/status_badge.dart 
// A StatelessWidget has no State object — it only knows what's passed in.
class StatusBadge extends StatelessWidget {
  final int statusCode;   // required input — this widget can't change it
  final bool small;       // optional input with a default

  const StatusBadge({super.key, required this.statusCode, this.small = false});

  // build() is called every time Flutter needs to draw this widget.
  // It returns the widget tree that describes what to show.
  @override
  Widget build(BuildContext context) {
    final color = statusColor(statusCode);   // green / amber / red
    final label = statusLabel(statusCode);   // "NORMAL" / "LOW" / "CRITICAL"

    return Container(
      // Containers wrap children and let you apply decoration
      padding: const EdgeInsets.symmetric(horizontal: 8, vertical: 3),
      decoration: BoxDecoration(
        color: color.withAlpha(30),
        borderRadius: BorderRadius.circular(20),
        border: Border.all(color: color.withAlpha(100), width: 1),
      ),
      child: Text(label,
        style: TextStyle(color: color, fontSize: 10, fontWeight: FontWeight.w700),
      ),
    );
  }
}
Why const constructor?

The const keyword tells Dart this object can be created at compile time. Flutter can reuse the same instance across rebuilds instead of creating a new one — a small but meaningful performance win.

A Real StatefulWidget — BedCard

BedCard needs an internal animation (the pulsing border when CRITICAL). Animations require an AnimationController, which must be stored somewhere that persists between rebuilds. That's what State is for.

// "with SingleTickerProviderStateMixin" gives us vsync for animations.
class _BedCardState extends State<BedCard> with SingleTickerProviderStateMixin {

    super.dispose();
  }

  @override

    // widget.config accesses the StatefulWidget's fields from State
    return GestureDetector(onTap: widget.onTap, child: ...);

}

Chapter 2

State — Data That Changes Over Time

State is any data that can change and that should update the UI when it does. The fluid level in a bed is state. Whether a bed is CRITICAL is state. The user's poll interval preference is state.

Key Concept: setState()

setState() is Flutter's built-in way to update local widget state. You call it, change the data inside, and Flutter re-runs build() with the new data. It's simple but limited to one widget — it can't share state across multiple screens.

The Problem setState() Doesn't Solve

Imagine the polling timer fires with a new reading for Bed 3. We need to update three places simultaneously:

  • The BedCard on the home screen (update the gauge)
  • The BedDetailScreen header (if that bed is currently open)
  • The CriticalBanner (add or remove bed 3 from the list)

With setState(), you'd have to pass a callback up through every parent widget, then back down to each child. This "prop drilling" becomes unmaintainable. That's why we use Riverpod — which is what Chapter 3 covers.

Implicit Animations with AnimatedContainer

Flutter has "implicit" animation widgets — they animate between old and new values automatically when you call setState(). No AnimationController needed. IV Sentinel uses AnimatedSwitcher to crossfade the status badge when a bed changes from NORMAL to LOW.

lib/features/home/widgets/bed_card.dart — AnimatedSwitcher 
// AnimatedSwitcher crossfades between its old and new child (300ms).
// The "key" is critical — when the key changes, it knows to animate.
AnimatedSwitcher(
  duration: const Duration(milliseconds: 300),
  child: reading != null
      ? StatusBadge(
          key: ValueKey(statusCode), // ← different key = animate
          statusCode: statusCode,
          small: true,
        )
      : const SizedBox.shrink(),   // empty box while loading
)
⚠️ The ValueKey rule

Without a ValueKey, AnimatedSwitcher can't tell whether you're showing a new widget or updating an existing one. Always give the child a key whose value changes when the content changes.

Explicit Animations — AnimationController

The CRITICAL border pulse is an explicit animation — you control it manually with an AnimationController. You start it, stop it, and define exactly how it moves with a Tween.

lib/features/home/widgets/bed_card.dart — pulse animation 
// AnimationController drives the animation forward/backward over time.
_pulse = AnimationController(
  vsync: this,                                   // "this" = the State (via mixin)
  duration: const Duration(milliseconds: 900),
);

// A Tween defines the range. Here: opacity pulses between 0.3 and 1.0.
_pulseAnim = Tween<double>(begin: 0.3, end: 1.0).animate(
  CurvedAnimation(parent: _pulse, curve: Curves.easeInOut),
);

// Start looping when bed enters CRITICAL; stop when it clears.
void _updatePulse() {
  if (widget.reading?.isCritical == true) {
    _pulse.repeat(reverse: true); // ping-pong: 0.3→1.0→0.3→...
  } else {
    _pulse.stop();
    _pulse.value = 0;
  }
}

// AnimatedBuilder listens to the controller and rebuilds only the border.
AnimatedBuilder(
  animation: _pulseAnim,
  builder: (context, child) => Container(
    decoration: BoxDecoration(
      border: Border.all(
        color: color.withAlpha((_pulseAnim.value * 220).toInt()),
      ),
    ),
    child: child, // ← child is NOT rebuilt on each animation tick
  ),
)
Why is child: child passed separately in AnimatedBuilder instead of building it inline?
A It's just a style preference — both approaches work the same way
B Flutter requires it — you'll get an error if you inline the child
C Performance: the child subtree is only built once and cached. Only the wrapper (Container with the border) rebuilds on each animation tick.
Chapter 3

Riverpod — Global State Without Prop Drilling

Riverpod is a reactive state management library. Think of it as a global key-value store where the values are live — any widget that reads a value is automatically rebuilt when the value changes.

Key Concepts

Provider — a named piece of global state. Declared once, readable anywhere.
ProviderScope — the root wrapper that initializes the store. Required at the top of every Riverpod app.
ConsumerWidget — a widget that can watch providers. Gets a ref parameter.
ref.watch() — subscribe: rebuild when the value changes.
ref.read() — read once without subscribing (use in callbacks).

The Provider Hierarchy in IV Sentinel

ProviderTypeHolds
appSettingsProviderStateNotifierProviderBed list, poll interval, toggles — persisted to device
bedReadingsProviderAsyncNotifierProviderLatest reading per bed, refreshed on a timer
bedHistoryProviderFutureProvider.family60-point history per bed (for the detail chart)

StateNotifier — The Bed Settings Store

AppSettingsNotifier stores the user's bed list and preferences. Any widget that watches appSettingsProvider rebuilds when beds are added, removed, or settings change.

lib/features/settings/bed_config_notifier.dart 
// StateNotifier<T>: holds state of type T. Notifies listeners when state changes.
class AppSettingsNotifier extends StateNotifier<AppSettings> {
  AppSettingsNotifier() : super(const AppSettings()) {
    _load(); // load persisted settings from device on startup
  }

  // "state = ..." triggers a rebuild for all watchers.
  // We NEVER mutate state directly — always replace it (immutable pattern).
  Future<void> addBed(BedConfig bed) async {
    state = state.copyWith(beds: [...state.beds, bed]);
    await _saveBeds(); // persist to shared_preferences
  }

  Future<void> removeBed(String id) async {
    state = state.copyWith(
      beds: state.beds.where((b) => b.id != id).toList(),
    );
    await _saveBeds();
  }
}

// The provider — this is the "address" widgets use to find the state.
final appSettingsProvider =
    StateNotifierProvider<AppSettingsNotifier, AppSettings>(
  (ref) => AppSettingsNotifier(),
);

Reading Providers in Widgets

lib/features/home/home_screen.dart — ConsumerWidget 
// ConsumerWidget gives us a "ref" — the key into the Riverpod store.
class HomeScreen extends ConsumerWidget {
  const HomeScreen({super.key});

  @override
  Widget build(BuildContext context, WidgetRef ref) {
    // ref.watch() — rebuild HomeScreen whenever appSettingsProvider changes.
    final settings = ref.watch(appSettingsProvider);

    // ref.watch() on an async provider — gives loading/error/data states.
    final readingsAsync = ref.watch(bedReadingsProvider);

    return Scaffold(
      body: readingsAsync.when(
        loading: () => LoadingGrid(count: settings.beds.length),
        error: (e, st) => ErrorView(message: e.toString()),
        data: (readings) => BedGrid(readings: readings),
      ),
    );
  }
}
watch() vs read() — when to use which

Use ref.watch() inside build() — it subscribes and rebuilds the widget when the value changes.
Use ref.read() inside callbacks (button presses, timers) — it reads once without subscribing. Using watch() inside a callback is a common mistake that causes bugs.

AsyncNotifier — Live Polling with Loading States

BedReadingsNotifier is an AsyncNotifier. Its state has three possible conditions — loading, error, or data — which map directly to the three branches of .when() in the UI.

lib/core/providers/bed_readings_provider.dart 
class BedReadingsNotifier extends AsyncNotifier<Map<String, BedReading>> {
  Timer? _timer;
  final Map<String, int> _prevStatus = {};

  @override
  Future<Map<String, BedReading>> build() async {
    ref.onDispose(() => _timer?.cancel());

    // .select() avoids rebuilding the timer when dark mode toggles, etc.
    final interval = ref.watch(
      appSettingsProvider.select((s) => s.pollIntervalSeconds),
    );

    // Timer fires _poll() every N seconds — the heartbeat of the app.
    _timer = Timer.periodic(Duration(seconds: interval), (_) => _poll());

    return _fetchAll(); // first fetch immediately
  }
}
A user taps "Refresh" on the home screen. The correct way to trigger a re-fetch is:
A ref.watch(bedReadingsProvider.notifier).refresh() — watch gives us the notifier
B ref.read(bedReadingsProvider.notifier).refresh() — read is correct inside a callback
C setState(() => {}) — this triggers a global rebuild
Chapter 4

Models & JSON — Structuring Your Data

A model class is a Dart class whose only job is to hold data. No UI, no network calls. Just fields and methods to convert to/from JSON for storage or transmission.

BedConfig — What You Configure

lib/core/models/bed_config.dart 
@immutable // tells Dart: once created, no fields can change
class BedConfig {
  final String id;            // unique ID — used as Riverpod family key
  final String name;          // e.g. "Bed 3A"
  final String channelId;     // ThingSpeak channel number
  final String apiKey;        // ThingSpeak read API key
  final double lowThreshold;  // default 30%
  final double critThreshold; // default 15%

  // copyWith lets you "update" an immutable object by creating a new one.
  // e.g. config.copyWith(name: "Bed 4") returns a new BedConfig with only name changed.
  BedConfig copyWith({String? name, String? channelId, ...}) => BedConfig(
    id: id, name: name ?? this.name, channelId: channelId ?? this.channelId, ...
  );

  // Serialize to JSON for SharedPreferences storage.
  Map<String, dynamic> toJson() => {
    'id': id, 'name': name, 'channelId': channelId, 'apiKey': apiKey,
    'lowThreshold': lowThreshold, 'critThreshold': critThreshold,
  };

  // Deserialize from JSON back into a BedConfig object.
  // "factory" = constructor that can return a new or existing instance.
  factory BedConfig.fromJson(Map<String, dynamic> json) => BedConfig(
    id: json['id'] as String,
    name: json['name'] as String,
    lowThreshold: (json['lowThreshold'] as num?)?.toDouble() ?? 30.0,
    ...
  );

  // == and hashCode are critical for FutureProvider.family cache lookups.
  // Without these, every navigation to BedDetailScreen fires a fresh HTTP request.
  @override
  bool operator ==(Object other) =>
      other is BedConfig && id == other.id && name == other.name && ...;

  @override
  int get hashCode => Object.hash(id, name, channelId, apiKey, lowThreshold, critThreshold);
}
Why override == and hashCode?

Dart's default == compares object identity (same address in memory), not value. Two BedConfig objects with identical fields would be "different" by default. FutureProvider.family uses == to decide if it has a cached result — without the override, every screen navigation triggers a new HTTP fetch even when nothing changed.

Why does BedConfig use copyWith() instead of just changing the fields directly?
A Because BedConfig is immutable (@immutable + final fields). You can't change fields after creation — you make a new object with the changed values.
B It's just a convenience helper — you could also change fields directly.
C Riverpod requires it — providers don't work with mutable objects.
Chapter 5

Async / HTTP — Talking to ThingSpeak

Network requests take time. Dart handles this with async/await — the same pattern as JavaScript's Promises. The app polls ThingSpeak every 15 seconds for every configured bed.

Key Concepts

Future<T> — a value that will be available at some point in the future. Like a promise.
async — marks a function as asynchronous (it returns a Future).
await — pauses until a Future completes, then gives you the value.

ThingSpeakService — Parsing Field1, Field2, Field3

lib/core/services/thingspeak_service.dart 
Future<BedReading> fetchLatest(String channelId, String apiKey) async {
  // Build the URL. ThingSpeak returns the last single reading as JSON.
  final uri = Uri.parse(
    'https://api.thingspeak.com/channels/$channelId/feeds/last.json?api_key=$apiKey',
  );

  // "await" pauses here until the HTTP response arrives (max 10s).
  final response = await http.get(uri).timeout(const Duration(seconds: 10));

  if (response.statusCode != 200) {
    throw Exception('ThingSpeak error ${response.statusCode}');
  }

  return _parseEntry(jsonDecode(response.body) as Map<String, dynamic>, channelId);
}

BedReading _parseEntry(Map<String, dynamic> entry, String fallbackBedId) {
  // field3 = bed ID stamped by the hardware. Fall back to channel ID if absent.
  final bedId = (entry['field3'] as String?)?.trim().isNotEmpty == true
      ? entry['field3'] as String
      : fallbackBedId;

  // double.tryParse returns null if the string isn't a valid number.
  // The ?? 0.0 provides a default so we never crash on bad data.
  final percent = double.tryParse(entry['field1'] as String? ?? '') ?? 0.0;
  final statusCode = int.tryParse(entry['field2'] as String? ?? '') ?? 0;
  final timestamp = DateTime.tryParse(entry['created_at'] as String? ?? '')
      ?? DateTime.now();

  return BedReading(
    bedId: bedId,
    percent: percent.clamp(0, 100),    // ensure 0–100 range
    statusCode: statusCode.clamp(0, 2),  // ensure valid 0/1/2
    timestamp: timestamp,
  );
}

Parallel Fetches — All Beds at Once

lib/core/providers/bed_readings_provider.dart — _fetchAll() 
Future<Map<String, BedReading>> _fetchAll() async {
  final beds = ref.read(appSettingsProvider).beds;
  if (beds.isEmpty) return {};

  // Preserve last known readings — if one bed fails, others aren't wiped.
  final result = Map<String, BedReading>.from(state.valueOrNull ?? {});

  // Future.wait runs all fetches concurrently (in parallel), not one-by-one.
  // With 4 beds at 15s timeout, sequential would take up to 60s. Parallel: 15s max.
  await Future.wait(
    beds.map((bed) async {
      try {
        result[bed.id] = await _service.fetchLatest(bed.channelId, bed.apiKey);
      } catch (_) {
        // This bed failed — keep its previous reading, don't throw.
      }
    }),
  );

  return result;
}
If you have 4 beds and each ThingSpeak call takes 2 seconds, how long does _fetchAll() take with Future.wait() vs sequential await?
A Same — Dart is single-threaded so they always take the same time
B Future.wait: ~2s (all run at once). Sequential await: ~8s (one after another)
C Sequential is faster — less overhead than managing parallel Futures

Navigation — Moving Between Screens

Flutter's Navigator manages a stack of screens (called "routes"). You push a new screen onto the stack to go forward, and pop to go back. IV Sentinel uses named routes for clean URL-style navigation.

MaterialApp(

    '/':         (_) => const HomeScreen(),

// Notification tap → navigate from OUTSIDE the widget tree:
navigatorKey.currentState?.popUntil((route) => route.isFirst);
navigatorKey.currentState?.pushNamed('/detail', arguments: bed);
Why navigatorKey?

Flutter navigation normally requires a BuildContext — which only exists inside widgets. When a notification tap fires, we're outside the widget tree. A GlobalKey<NavigatorState> gives us a handle to the Navigator that's accessible from anywhere, including the AlertService callback.

Custom Painting — The Fluid Gauge

Standard Flutter widgets can't draw arbitrary shapes. For the circular arc gauge, we use CustomPainter — a direct API to the Skia drawing canvas. This is how Apple Watch rings, Oura readiness scores, and most premium gauges are built.

Key Concept: CustomPainter

CustomPainter gives you a Canvas object and a Size. You draw on the canvas using Paint objects (configure color, stroke, etc.) and drawing commands (drawArc, drawCircle, drawLine, etc.).

The Arc Geometry

Flutter's drawArc measures angles from the 3 o'clock position, going clockwise (because screen y-axis points down). Our gauge:

  • startAngle: π × 0.75 (135°) = 7:30 position (bottom-left)
  • sweepAngle: π × 1.5 (270°) = clockwise to 4:30 position (bottom-right)
  • Gap at the bottom between 4:30 and 7:30 = empty space that looks like a dial
lib/features/home/widgets/fluid_gauge.dart — painter 
class _GaugePainter extends CustomPainter {
  final double percent;   // 0–100 (already animated by TweenAnimationBuilder)
  final Color color;     // green / amber / red based on status
  final Color trackColor;
  final double strokeWidth;

  static const _startAngle = math.pi * 0.75; // 135° from 3 o'clock
  static const _totalSweep = math.pi * 1.5;  // 270° total arc

  @override
  void paint(Canvas canvas, Size size) {
    final center = Offset(size.width / 2, size.height / 2);
    final radius = (size.width - strokeWidth) / 2;
    final rect = Rect.fromCircle(center: center, radius: radius);

    // 1. Background track — full 270° in dim color
    final trackPaint = Paint()
      ..style = PaintingStyle.stroke
      ..strokeWidth = strokeWidth
      ..strokeCap = StrokeCap.round
      ..color = trackColor;
    canvas.drawArc(rect, _startAngle, _totalSweep, false, trackPaint);

    // 2. Filled arc — only (percent/100) of the total sweep
    final fillSweep = _totalSweep * (percent / 100);
    final fillPaint = Paint()
      ..style = PaintingStyle.stroke
      ..strokeWidth = strokeWidth
      ..strokeCap = StrokeCap.round
      ..color = color;
    canvas.drawArc(rect, _startAngle, fillSweep, false, fillPaint);

    // 3. Tip dot — bright circle at the leading edge of the fill
    final tipAngle = _startAngle + fillSweep;
    final tipX = center.dx + radius * math.cos(tipAngle);
    final tipY = center.dy + radius * math.sin(tipAngle);
    canvas.drawCircle(Offset(tipX, tipY), strokeWidth * 0.45,
      Paint()..color = color);
  }

  // shouldRepaint: return true only when relevant data changed.
  // Returning true always works but wastes performance.
  @override
  bool shouldRepaint(_GaugePainter old) =>
      old.percent != percent || old.color != color;
}

Charts with fl_chart

IV Sentinel has two charts: a tiny sparkline on each bed card (no axes, purely visual), and a full history chart on the detail screen with labeled axes, dashed threshold lines, and a touch tooltip. Both use the fl_chart package.

Key Concept: fl_chart data model

FlSpot(x, y) — a single data point. X is the index (0, 1, 2…), Y is the value (0–100).
LineChartBarData — one line on the chart. Holds a list of FlSpots + visual config.
LineChartData — the full chart config: axes, grid, data lines, touch behavior.
ExtraLinesData — horizontal/vertical reference lines drawn on top of everything (used for threshold lines).

Sparkline — BedCard Mini Chart

The sparkline shows the last 20 readings as a small trendline at the bottom of each bed card. No labels, no axes — just the shape of the curve in the status color.

lib/features/home/widgets/sparkline_chart.dart 
class SparklineChart extends StatelessWidget {
  final List<double> readings;
  final int statusCode;
  const SparklineChart({super.key, required this.readings, required this.statusCode});

  @override
  Widget build(BuildContext context) {
    final color = statusColor(statusCode);

    // Convert raw percent values to FlSpot(index, value)
    final spots = [
      for (var i = 0; i < readings.length; i++)
        FlSpot(i.toDouble(), readings[i]),
    ];

    return LineChart(
      LineChartData(
        minY: 0, maxY: 100,
        // No axes, no grid — pure visual sparkline
        titlesData: const FlTitlesData(show: false),
        gridData: const FlGridData(show: false),
        borderData: FlBorderData(show: false),
        lineTouchData: const LineTouchData(enabled: false),
        lineBarsData: [
          LineChartBarData(
            spots: spots,
            isCurved: true,
            curveSmoothness: 0.35,
            color: color,
            barWidth: 1.5,
            dotData: const FlDotData(show: false),
            // Gradient fill below the line — makes it feel like a metric
            belowBarData: BarAreaData(
              show: true,
              gradient: LinearGradient(
                begin: Alignment.topCenter,
                end: Alignment.bottomCenter,
                colors: [color.withAlpha(60), color.withAlpha(0)],
              ),
            ),
          ),
        ],
      ),
      duration: const Duration(milliseconds: 200),
    );
  }

The sparkline disables touch with LineTouchData(enabled: false). Why, given that the detail screen's chart has touch enabled?
A The sparkline is 28px tall and lives inside a GestureDetector (the BedCard tap). Enabling touch would intercept card taps without giving enough space to be useful.
B fl_chart requires touch to be disabled on StatelessWidgets.
C Touch would conflict with the AnimatedBuilder driving the CRITICAL border pulse.
Chapter 9

Hero Animations — Shared-Element Transitions

A Hero animation is when an element visually flies from one screen to another. In IV Sentinel, the FluidGauge on the bed card grows and flies into the header of the detail screen when you tap a card. This is one of the most premium-feeling interactions in mobile apps.

Key Concept: Hero widget

Wrap the same widget on both screens with Hero(tag: 'same-string'). Flutter automatically animates between them during navigation. The tag must be unique across all currently visible Heroes.

Step 1 — Tag the gauge on the BedCard

lib/features/home/widgets/bed_card.dart 
// Hero tag uses the bed ID — unique per card so multiple heroes don't conflict.
Hero(
  tag: 'gauge-${widget.config.id}',
  child: FluidGauge(
    percent: reading?.percent ?? 0,
    statusCode: statusCode,
    isLoading: reading == null,
    size: 100,
  ),
)

Step 2 — Match the tag on the detail screen

lib/features/detail/bed_detail_screen.dart — _GradientHeader 
// Same tag as the card — Flutter connects the two and animates between them.
// The gauge grows from 100px (card) to 140px (detail header) during the transition.
Hero(
  tag: 'gauge-${config.id}',
  child: FluidGauge(
    percent: percent,
    statusCode: statusCode,
    size: 140,  // larger on the detail screen
  ),
)
Hero animation is free

You don't write any animation code. Flutter's navigation framework detects matching Hero tags, clips the outgoing screen, and morphs the widget between its two positions. The only requirement: the widget must be a StatelessWidget or have stable state between screens.

The Gradient Detail Header

The header uses extendBodyBehindAppBar: true and a transparent AppBar so the gradient can bleed all the way to the top of the screen including under the status bar. The gradient tints the status color subtly at the top and fades to the surface color at the bottom.

You have 4 bed cards all on screen at once. Each one has a Hero with tag 'gauge-${config.id}'. What happens if two beds have the same ID?
A Flutter throws an assertion error: "There are multiple heroes that share the same tag." Tags must be unique per screen.
B Flutter picks one of them arbitrarily and ignores the other.
C The animation plays twice, once for each matching Hero.
